目标



规范view（page）的编写，使其能被通用框架加载并在各种场景下适配显示。

一个简单对象


{
    name: 'helloworld',  // 唯一标识
    title: '你好',    // 标题
    buttons: [...],   // 导航按钮，包括回退按钮和右侧按钮
    startup: function() {
        // 划入或加载时的入口方法
    },
    teardown: function() {
        // 划出或卸载时的入口方法
    }    
}

导航的类型




标题格式


title: 'pure text'
or
title: '<div>a html fragment</div>'

按钮的属性和行为


 id - 唯一标识
 type - 标注是回退按钮(back)还是右侧按钮(func)
 text - 文本（可选）
 bg - 背景图片（可选）
 icon - 图标（可选
 handler - 点击句柄（回退按钮默认为history.back
 hide - 是否隐藏

回退按钮只存在一个，右侧按钮可存在多个

view（page）的操作



工具方法和属性，可在上下文中被访问

导航相关操作



    前进：open(url) or push(fragment)
    后退：back() or pop()
    设置标题：setTitle(title)
    设置按钮：setButton(options)

内容相关操作


    填充内容：fill(html)
    内容区域结点：el
    内容区域的$结点：$el

问题和困难


view(page)是个最小子集 or 严格定义？

业务代码中依赖某特通用框架的模块或方法，在其他通用框架中是不被支持的。

在业务场景的压力下，也许view(page)的定义需要快速迭代
